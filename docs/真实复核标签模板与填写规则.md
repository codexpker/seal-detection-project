# 真实复核标签模板与填写规则

## 1. 模板位置

- 当前待复核模板：`reports/lab_review_label_template_v1_run1/review_labels_template.csv`
- 生成脚本：`src/scripts/lab_review_label_template_v1.py`

## 2. 真正需要填写的字段

回灌脚本 [lab_health_bank_feedback_loop_v1.py](/Users/xpker/seal_detection_project/src/scripts/lab_health_bank_feedback_loop_v1.py) 只会读取这四列：

- `run_id`
- `review_label`
- `reviewer`
- `review_note`

模板里的其他列都是辅助上下文，用来帮助你判断，不参与回灌逻辑。

## 3. 推荐标签

请尽量直接使用这 3 个标准值，不要自己扩写：

- `healthy`
  - 含义：确认该运行可以视为健康/正常参考。
  - 当前项目里通常对应：`sealed`、无明显泄漏证据、可作为健康窗口库候选。

- `anomaly`
  - 含义：确认该运行可以视为异常/泄漏/非密封参考。
  - 当前项目里通常对应：`unsealed`、明确状态转移异常、明确静态异常。

- `uncertain`
  - 含义：当前不下结论，只保留为已复核但不回灌健康库。
  - 当前项目里通常对应：`breathing_watch`、强干扰、证据不够闭合的难例。

## 4. 不同状态的填写建议

- `transition_boost_alert`
  - 如果确认是真实密封到非密封的状态转移，填 `anomaly`
  - 如果转移时间不可信或事件解释不闭合，填 `uncertain`

- `static_dynamic_supported_alert`
  - 如果结合实验记录确认是不密封/异常，填 `anomaly`
  - 如果后看发现只是静态漂移或背景波动，填 `uncertain`

- `static_hard_case_watch`
  - 如果确认是密封但有材料呼吸/释湿效应，填 `healthy`
  - 如果确认确实存在泄漏证据，填 `anomaly`
  - 如果仍然拿不准，填 `uncertain`

- `static_abstain_low_signal`
  - 如果运行本身可确认是正常密封，可填 `healthy`
  - 如果标签来源不稳定或证据太弱，优先填 `uncertain`

## 5. `review_note` 怎么写

建议固定写成一句短因果，不要只写“已看过”。

推荐格式：

- `实验记录确认密封正常，可作为健康参考`
- `确认 2026-03-04 16:39 开孔后转为非密封`
- `疑似胶泥呼吸效应，暂不回灌健康库`
- `静态证据不足，先保留 uncertain`

## 6. 当前这批运行的优先级

优先按模板里的 `review_priority` 来填：

- `P0_transition`
  - 最先确认
  - 因为这批最直接影响现场演示强证据

- `P1_static_alert`
  - 第二优先级
  - 用来确认静态异常是否真的成立

- `P2_hard_case_watch`
  - 第三优先级
  - 用来确认是否属于密封难例/呼吸效应

填完后，直接跑：

```bash

  --labels-csv reports/lab_review_label_template_v1_run1/review_labels_template.csv \

  --window-table-csv reports/lab_unified_data_interface_v1_run4/window_table.csv \
  --review-output-csv reports/lab_unified_data_interface_v1_run4/review_output.csv \
  --output-dir reports/lab_health_bank_feedback_loop_v1_run2
```
