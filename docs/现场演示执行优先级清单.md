# 现场演示执行优先级清单

## 1. 目的

这份清单用于把 [试验方案-v1.xlsx](/Users/xpker/Library/Containers/com.tencent.xinWeChat/Data/Documents/xwechat_files/wxid_g1sxmmmhath522_e268/msg/file/2026-03/试验方案-v1.xlsx) 重排成“当前系统真正能支撑验证和演示”的执行版本。

当前必须严格对齐项目主路线：

`分支感知路由 -> 转移段相对打分 -> 状态段级证据融合 v2`

因此本清单不追求覆盖全部试验构想，而只保留当前已经有离线证据支撑、且适合现场讲解的工况。

---

## 2. 当前可演示能力边界

当前系统已经能够稳定支撑的能力只有三类：

1. `高湿 + 密封 -> 不密封` 状态切换的强证据检测
2. `高湿静态段` 下的辅助风险判断
3. 对难例和强干扰场景输出 `watch / abstain`，而不是强行误报

当前**不能**作为现场已解决能力对外表达的内容：

1. 全局统一 `seal / unseal` 分类
2. 加水后液态水进入场景的稳定建模
3. 现场长期监测的异常闭环

---

## 3. 执行优先级

## P0 必须执行

### 工况 A：电机1 / 高湿 / 密封 -> 不密封切换

- 目标：
  验证当前最强主线，即 `transition_boost_alert`
- 对应试验方案：
  - `3-30 17:00 ~ 3-31 09:00` 不启动，高湿
  - `3-31 09:00 ~ 13:00` 启动，高湿
  - `3-31 13:00 ~ 17:00` 不启动，高湿
  - `2026-03-31 17:00` 密封 -> 不密封切换
  - `3-31 17:00 ~ 4-1 09:00` 不启动，高湿
  - `4-1 09:00 ~ 13:00` 启动，高湿
  - `4-1 13:00 ~ 17:00` 不启动，高湿
- 当前期待输出：
  - 切换后进入 `transition_boost_alert`
  - 关键证据包含 `delta_in_hum`、`delta_half_in_hum`、`max_hourly_hum_rise`、`corr_AH`
- 现场价值：
  这是当前最能证明“系统确实抓到了密封状态变化”的场景

### 工况 B：电机1 / 高湿静态对照

- 目标：
  展示系统在高湿静态段能够给出辅助风险，而不是只会抓切换点
- 重点时段：
  - 切换前夜间高湿静态段
  - 切换后夜间高湿静态段
- 当前期待输出：
  - `seal` 段可能是 `static_abstain_low_signal` 或 `static_low_risk`
  - `unseal` 段有机会进入 `static_dynamic_supported_alert`
- 现场价值：
  用来补充说明“静态不是完全没信息，但系统不会夸大静态能力”

### 工况 C：难例 / 保守输出展示

- 目标：
  展示系统不会乱报
- 建议直接用已有演示样例：
  - `2026-03-06 160246_seal_unheated` 对应 `static_hard_case_watch`
  - `2026-03-02 161335_seal_unheated` 对应 `static_abstain_low_signal`
- 当前期待输出：
  - `watch`
  - `abstain`
- 现场价值：
  这是系统工程可信度的关键，不应省略

## P1 可选执行

### 工况 D：电机1 / 高湿下启动与停机后的恢复段

- 目标：
  观察热相关段是否会被系统谨慎处理，而不是误报为泄漏
- 对应试验方案：
  - `3-31 09:00 ~ 13:00`
  - `3-31 13:00 ~ 17:00`
  - `4-1 09:00 ~ 13:00`
  - `4-1 13:00 ~ 17:00`
- 当前正确讲法：
  - 这些段主要用于验证路由和保守性
  - 不应当成当前主告警证据
- 现场价值：
  如果时间允许，可以补一句“系统能识别热相关干扰，不会把所有波动都解释成密封异常”

---

## P2 暂不作为现场验收项

### 工况 E：电机2 / 加水模拟泄漏

- 不作为当前主演示项
- 原因：
  - 这类场景更像“液态水进入后的滞留 / 蒸发 / 排出”问题
  - 当前主链路不是围绕这类物理过程建立的
  - 现在拿它做主演示，风险大于收益
- 当前定位：
  后续研究工况

### 工况 F：电机3、4 / 现场长期监测

- 不作为当前“已验证异常检测能力”的展示内容
- 原因：
  - 这部分当前更适合作为基准数据积累和后续相似性库建设
  - 不能在现场前几天把它包装成已经闭环验证的能力
- 当前定位：
  后续现场迁移准备项

## 4. 现场执行顺序

建议严格按下面顺序执行，不要跳：

1. 先完成 `电机1 / 高湿 / 密封段`
2. 在 `2026-03-31 17:00` 做明确切换，并记录绝对时间
3. 继续完成 `电机1 / 高湿 / 不密封段`
4. 用演示脚本生成总览和重点卡片
5. 现场只讲：
   - 切换强证据
   - 静态辅助证据
   - 难例保守输出

## 5. 每次试验后的固定输出

每次新试验结束后，固定只看这几项：

1. `final_status`
2. `risk_level`
3. `primary_evidence`
4. `primary_segment_id`
5. 关键证据段的时间范围

固定使用这两个入口：

```bash
MPLCONFIGDIR=/tmp/mplconfig python src/scripts/lab_demo_entry_v1.py --output-dir reports/lab_demo_entry_v1_run1
MPLCONFIGDIR=/tmp/mplconfig python src/scripts/lab_interference_regression_entry_v1.py --output-dir reports/lab_interference_regression_entry_v1_run1
```

## 6. 现场汇报口径

只能按下面口径讲：

- 我们当前已经验证：高湿切换场景下，系统能稳定给出强证据
- 静态高湿场景下，系统能给出辅助风险，但不承诺全覆盖
- 对难例和强干扰，系统会主动 `watch / abstain`

不能按下面口径讲：

- 已经解决所有密封 / 不密封判别
- 所有工况都能稳定分类
- 现场长期数据已经完成异常检测闭环

---

## 7. 当前结论

如果只剩现场前最后一轮推进时间，执行重点应收缩为：

1. `电机1 / 高湿 / 密封 -> 不密封切换`
2. `电机1 / 高湿静态段`
3. 难例与干扰保守展示

其余工况保留研究价值，但不进入当前现场演示主链路。
