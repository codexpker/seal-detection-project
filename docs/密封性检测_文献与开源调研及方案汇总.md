# 电机接线盒密封性检测：文献、开源与路线重构

> 版本定位：实验室可分性优先版  
> 当前目标：不是先做一个“全局统一的密封/非密封分类器”，而是先证明哪些实验室工况里确实存在可重复利用的规律，再决定模型复杂度与现场迁移方式。

---

## 1. 问题重述

### 1.1 当前真实难点

这项工作当前卡住的根因，不是“模型还不够大”，而是下面这几个事实同时成立：

- 密封与非密封在**多数滑动窗口**上并没有稳定区分度。
- 仅有四路输入：`Tin`、`Tout`、`Hin`、`Hout` 与时间戳，观测维度有限。
- 影响因素耦合严重：外部湿度、热源开停、材料吸放湿、呼吸效应、状态切换时滞。
- 当前最可信的标签粒度更接近**文件级或状态段级**，不是密集窗口级标签。
- 实验室与现场存在明显分布迁移，实验室学到的“标准模式”不能直接假定可迁移。

### 1.2 因此需要修正的问题定义

近阶段问题应改写为：

1. 哪些工况本身就是低信息工况，应该直接排除出异常训练集？
2. 哪些工况或状态转移段存在稳定的可分性证据？
3. 在这些高信息工况内，哪类模型分支最有效？
4. 当且仅当多个分支存在互补性时，才讨论融合层，最后才讨论 MoE 或复杂集成。

### 1.3 近阶段不做什么

- 不以“实验室大部分工况统一建模成功”为近期目标。
- 不把现场迁移作为这一步的成败标准。
- 不因为单一分支表现不稳，就直接跳到“更大的模型”或“全局 MoE”。

---

## 2. 已有负结果证据

以下结论来自当前阶段汇报中已经展示过的实验现象，说明“全局统一分类边界”并不存在，或者至少当前还没有足够证据支持它存在。

### 2.1 低信息工况已经被反复验证

- **外部环境稳定 - 内部无热源**：密封与不密封整体变化趋势接近，差异不显著。
- **外部环境稳定 - 内部有热源**：升温后内部相对湿度会出现短时尖峰，但密封与不密封整体仍难区分。
- **内部积湿 - 有/无热源**：曲线可能持续高湿，但切换后未必出现清晰突变，难直接作为密封判别依据。

这些结果说明：

- 部分工况的主要价值不是“做分类”，而是帮助我们识别**哪些窗口不该送进异常模型**。
- 如果不先排除这些窗口，模型很容易学到伪相关或背景噪声。

### 2.2 目前看到的高价值线索是“特定工况/特定阶段”

- **外部高湿驱动 - 内部无热源**：不密封后内部湿度缓慢抬升，这是一类有物理解释的候选有效信号。
- **外部高湿背景 + 热源停止后的冷却段**：降温收缩形成负压，外部高湿空气进入内部，属于状态变化更明显的阶段。
- **中途打孔 / 密封转非密封**：这是最适合验证“状态转移附近分数是否持续抬升”的场景。

### 2.3 当前应接受的结论

当前最重要的结论不是“模型失败了”，而是：

- **多数窗口不可分** 是已知事实，不是训练技巧问题。
- 近阶段必须先回答“哪里可分”，再回答“怎么建模”。
- “低信息工况排除 + 高信息工况突破 + 复杂耦合工况延后”才是合理路线。

## 3. 工况分层与目标层级

## 3.1 三类工况角色

| 工况角色 | 当前代表场景 | 当前定位 | 近期处理策略 |
|---|---|---|---|
| 低信息工况 | 外部环境稳定-无热源；外部环境稳定-有热源 | 显式排除集 | 仅标记或 gate 掉，不进入异常训练 |
| 高信息工况 | 外部高湿-无热源；冷却降温段；中途打孔转移段 | 首批主战场 | 先证明可分性，再做分支模型 |
| 复杂耦合工况 | 外部波动明显、热湿多因素叠加 | 后续难点集 | 暂不作为首轮成败标准，后续再做深度模型验证 |

## 3.2 任务层级固定为四层

1. **信息窗口筛选**  
   先排掉明显低信息窗口，保留可能有物理信息的候选窗口。

2. **工况识别 / 粗分层**  
   按窗口所属工况族或状态段做条件化分析，而不是全局混合建模。

3. **工况内风险建模**  
   只在高信息工况内比较判别分支、残差分支、相似性分支。

4. **事件级融合判定**  
   最后再做迟滞、持续时间、冷却和事件级输出；融合层默认从轻量模型开始。

## 3.3 当前成功标准

当前“做出来”的定义不是覆盖所有工况，而是满足下面两点：

- 在**少数高信息工况**下，能稳定区分或稳定排序出非密封风险。
- 在**状态转移场景**下，异常分数能在转移邻域附近持续抬升，而不是只出现孤立尖峰。

## 4. 文献与开源地图

## 4.1 检索顺序必须固定

先找三类材料，再找更复杂模型：

1. 为什么多数窗口不可分  
2. 怎样筛信息窗口、怎样做工况/阶段分层  
3. 怎样做事件级评估与可靠验证  
4. 最后才是更强模型或更复杂融合

这个顺序不能反过来。

## 4.2 窗口筛选 / 分段 / 特征抽取

1. [ruptures](https://github.com/deepcharles/ruptures)  
   用途：做变化点检测、状态切换定位、阶段切分。  
   对当前问题的价值：比“整段统一滑窗”更适合处理中途打孔、热源开停、冷却段切换。

2. [STUMPY / Matrix Profile](https://stumpy.readthedocs.io/en/latest/Tutorial_The_Matrix_Profile.html)  
   用途：做 motif/discord 搜索、异常片段和相似窗口检索。  
   对当前问题的价值：适合做“健康窗口库”与“相似性分支”的基础设施，而不是先上复杂深度检索模型。

3. [tsfresh](https://tsfresh.readthedocs.io/en/main/)  
   用途：系统化提取统计、趋势、波动、相关性等时间序列特征。  
   对当前问题的价值：用于先构建可解释的规则 + XGBoost/LightGBM 分支，而不是一开始端到端黑盒。

## 4.3 基准与评估

1. [TSB-AD](https://github.com/TheDatumOrg/TSB-AD)  
   用途：提供更严格的时序异常检测评估视角。  
   对当前问题的启发：先把评估做可靠，再谈模型优劣。

2. [TSB-AutoAD](https://github.com/TheDatumOrg/TSB-AutoAD)  
   用途：自动化时序异常检测 benchmark。  
   对当前问题的启发：自动化集成或复杂配置并不天然优于简单路线，尤其在分布迁移下更要谨慎。

3. [Merlion](https://github.com/salesforce/Merlion)  
   用途：提供统一的基线、集成与评估工具。  
   对当前问题的价值：更适合做离线横评和对照，不适合作为当前问题的直接答案。

4. [Do Deep Neural Networks Contribute to Time Series Anomaly Detection?](https://www.sciencedirect.com/science/article/pii/S0950705122010094)  
   用途：审视深度模型在时序异常任务中的真实增益。  
   对当前问题的启发：当数据定义和评估口径不稳时，盲目上更深模型通常不会先解决根因。

## 4.4 深度分支代表方法

1. [TranAD](https://github.com/imperial-qore/TranAD)  
   用途：多变量时序异常检测代表性深度基线。  
   当前角色：作为残差/深度分支的对照组，而不是默认主路线。

2. [Anomaly Transformer](https://openreview.net/forum?id=LzQQ89U1qm_)  
   用途：基于 attention 的异常关联建模。  
   当前角色：适合后续复杂耦合工况做代表性深度对照，不适合替代前面的工况筛选和窗口筛选。

## 4.5 结合当前问题后的文献结论

1. 当前应优先补的是**可分性证据链**，不是更大的模型清单。
2. 当前最有工程价值的路线是：**工况分层 + 可解释基线 + 状态转移验证**。
3. MoE、stacking、boosting 可以保留，但必须放在“至少两个分支已有互补性证据”之后。

## 4.6 当前落地状态说明

为避免“文献里提到的都已经工程接入”的误解，这里明确区分三种状态：

1. **已直接体现在当前工程脚本中的方法**
   - `XGBoost` 判别支线：已实现并完成离线验证，但当前未通过验收。
   - 相似性 / 记忆分支：已实现并完成离线验证，当前已达到支线通过标准。
   - 分支感知路由、转移段相对打分、静态阈值基线：已实现并作为当前主路线的一部分。

2. **已体现在方案思想中，但尚未直接接入对应开源库**
   - `TSB-AD / TSB-AutoAD`：主要体现在评估口径上，例如 `leave-one-run-out`、避免相邻窗口泄漏、先做可靠验证再比较模型。
   - `STUMPY / Matrix Profile`：主要体现在当前“相似性 / 记忆分支”的设计思想上，但目前仍是自实现的轻量记忆检索，不是直接调用 `STUMPY`。
   - `tsfresh`：主要体现在当前先使用可解释统计量、斜率、差分、波动等特征构建规则和判别支线，而不是直接调用 `tsfresh` 做全量特征抽取。
   - `ruptures`：主要体现在当前优先重视状态切换、阶段切分和转移邻域验证的路线设计上，但还没有正式接入 `ruptures` 做变化点检测。

3. **当前仍作为后续储备方法，尚未进入主链路**
   - `Merlion`：保留作后续离线横评和基线对照工具。
   - `TranAD`、`Anomaly Transformer`：保留作残差/深度分支的代表性对照方法，目前未进入当前主链路。

因此，第 4 节当前的作用是：**给路线提供方法论锚点和实现优先级，而不是要求所有开源项目都已经接入代码。**

## 5. 实验室优先路线

## 5.1 第一阶段：先证明“哪里可分”

第一阶段的核心问题只有一个：

**在实验室数据里，哪些工况族或状态转移段是可利用的？**

第一阶段固定验证三类场景：

1. **低信息对照场景**  
   - 外部环境稳定-无热源  
   - 外部环境稳定-有热源  
   预期结果：不是分类成功，而是被 gate 掉或仅作标记。

2. **高价值场景**  
   - 外部高湿-无热源  
   预期结果：内部湿度缓慢抬升成为第一类可重复利用信号。

3. **状态转移场景**  
   - 中途打孔 / 密封转非密封  
   预期结果：分数在转移邻域附近持续抬升，而不是仅出现孤立尖峰。

## 5.2 第二阶段：只做受控的三分支基线

只有在第一阶段找到至少一个高信息工况族之后，才进入三分支比较：

1. **判别分支**  
   规则 + `XGBoost/LightGBM`  
   输入来自工况内特征、差分特征、斜率、波动、阶段统计量。

2. **残差分支**  
   正常样本预测残差  
   现有 `GRU` 路线保留，但只作为一个分支。后续可加轻量 `TCN` 或代表性深度基线做对照。

3. **相似性 / 记忆分支**  
   健康窗口库 + 相似窗口检索  
   优先从 Matrix Profile 或原型库做起，不先做复杂神经记忆网络。

### 当前进展对齐

基于当前仓库内的离线验收结果，第二阶段需要按下面的事实继续推进：

1. **第一阶段当前结论是 `PARTIAL_PASS`，不是 `PASS`**  
   - 工况 gate 已通过，说明“先筛工况再分析”的路线成立。  
   - 转移分支已通过，说明“中途打孔 / 密封转非密封”场景能做事件级抬分验证。  
   - 静态高信息分支尚未通过，说明“高外湿无热源下静态 seal/unseal 稳定区分”仍未建立充分证据。

2. **判别分支已经跑过第一轮 XGBoost 对照，但当前结果不支持把它作为主推荐**  
   - `XGBoost` 在当前 `leave-one-run-out` 验证中未超过简单基线。  
   - 当前更稳的静态替代是单特征阈值基线 `delta_half_dAH`，可先作为静态分支的演示版方案。  
   - 因此当前主线不应表述为“XGBoost 已解决静态分支”，而应表述为“XGBoost 支线已完成验证，但暂未通过验收”。

3. **当前实际推荐路线应更新为**  
   `分支感知路由 -> 转移段相对打分 -> 状态段级证据融合 v3`
   - 其中转移场景已升级为 `transition_boost_alert`。
   - 静态场景当前采用 `delta_half_dAH 阈值基线 + 相似性分支 + 多视角证据增强`。
   - 静态难例当前必须允许进入 `hard_case_watch / abstain`，而不是强行输出告警或低风险。

4. **相似性分支已进入当前运行级决策层，但不等于“已经进入模型融合层”**  
   - 相似性 / 记忆分支已经完成首轮验证，并达到当前支线通过标准。  
   - 它当前作为 `evidence fuser v2` 的一条静态证据源参与运行级决策，但这仍然是规则化证据融合，不是 `Logistic Regression / LightGBM / stacking` 意义上的模型融合。  
   - 残差分支仍未完成同等强度验证，在没有新证据前，不应替代当前已验证成立的主路线。

5. **第一步当前需要从“工况 gate”升级为“分支感知路由”**  
   - 当前已完成 `gate / info selector v2` 验证，结论是：第一步优化有效，但优化重点不是继续缩窗口，而是把窗口送到正确分支。  
   - 转移邻域窗口应完整路由到转移分支；当前 `transition_near_branch_coverage = 1.0`。  
   - 静态阈值分支不应强行覆盖所有静态高信息运行，而应只在 `delta_half_dAH` 为正等 favored 窗口上工作；当前该策略在覆盖 `5` 个运行时，把 run 级 `balanced_accuracy` 从 `0.6667` 提升到 `0.8333`。  
   - 这意味着新的第一步不是单纯 “gate 掉低信息窗口”，而是要允许不同后续分支看到不同的候选窗口集合，并允许低把握度分支显式 abstain。
   - 在此基础上，当前又完成了 `gate / info selector v3` 的首轮优化验证：  
     - `transition` 分数升级为多视角版本，近转移窗口平均排序从 `0.903` 提升到 `0.942`；平均分数抬升从 `0.804` 提升到 `2.638`。  
     - `static` 分支当前没有被强行重构，仍保留 `delta_half_dAH >= 0` 的 favored 规则；`v3` 只增加诊断性的 `static_context_score_v3` 和路由理由字段。  
     - 随后又完成了 `evidence fuser v3` 的接入验证：运行级 `final_status / primary_evidence / risk_level` 与 `v2` 完全一致，没有引入回退；同时 `transition` 的 rank / lift 解释被真正带到了运行级输出层。  
     - 在此基础上又完成了 `evidence fuser v4` 的保守接入验证：只对 `外部高湿-无热源` 静态支线叠加 `no_heat probe v3`，并且仅把 `2026-03-03 181049_unseal_unheated` 这一类 `multiview support` 样例升级成更高置信的 `supported alert`；其余分支和全部 transition 运行都保持不变。  
     - 这说明当前默认路径已经可以从 `v3` 进一步升级为 `v4`，后续优化仍应继续围绕 `transition event` 的边界与展示展开，而不是重新回到全局静态分类器。

6. **历史旧数据已确认不适合直接并入主训练集**  
   - 旧数据中存在一批 `strict sealed` 运行，会被当前规则长时间打成 `candidate_high_info`，说明它们会对主任务形成强干扰。  
   - 当前更合理的角色是：`干扰挑战集 + 负控制集 + 动态响应特征发现集`。  
   - 因此旧数据后续应主要用于验证系统是否能 `watch / abstain`，而不是直接增加 `seal/unseal` 监督样本。

7. **`dynamic response probe v1` 已完成首轮验证，但当前定位应是辅助探针**  
   - 在旧数据中，`best_lag_rh_h`、`gain_ratio_dAH_change` 等动态响应特征对 `unsealed vs sealed_strict` 已表现出一定区分能力。  
   - 但在当前分工况小样本中，这些特征方向只部分一致，还不足以替代现有主路线。  
   - 因此这条线当前更适合作为 `interference/watch` 辅助探针，而不是直接升级成新的主判别分支。
   - 进一步把 `外部高湿驱动` 显式拆成 `无热源响应` 和 `冷却响应` 两条分支后，当前结果也更清楚了：  
     - `外部高湿-无热源` 这条物理逻辑并不是“筛不出来”，而是已经能把 `低信号 sealed`、`response-supported unsealed` 和 `response-like sealed hard case` 三种状态分开。  
     - 当前最典型的干扰不是“没有响应”，而是存在 `sealed 但表现出外部驱动样` 的难例，这更像 `材料吸放湿 / 残余湿气 / 结构呼吸效应` 对主信号的污染。  
     - `外部高湿-冷却响应` 当前只在 `unsealed heated` 运行里提取到了明确候选段，但尚缺少足够的 `sealed heated cooling` 对照窗口，因此现在只能作为“理论成立、待后续验证”的显式分支，不能直接升成已验收主能力。
   - 在此基础上进一步验证了 `2h 短窗增强 + 6h 主判定 + 12h 长窗确认` 的谨慎多尺度方案：  
     - 对 `外部高湿-无热源`，当前更合理的落法是 `6h` 做主判定、`12h` 做持续性确认、`2h` 只做局部响应增强；短窗不能直接主导最终结论。  
     - 对 `外部高湿-冷却响应`，`12h` 的累计确认比 `2h` 更有价值；当前短窗没有证明自己能带来额外稳定判别力。  
     - 因此，多尺度 `v2` 当前已经接入 `review_output` 和演示卡片，作为显式补充证据；但它尚未替换全局默认判定链。
   - 围绕主战场 `外部高湿-无热源`，又新增了 `no_heat probe v3`，把这条线进一步收敛成 `早段响应 + 晚段持续性` 的三态解释：  
     - `negative`：早段里外部 RH 虽然上升，但内部 RH 不跟随，属于 `drive-decoupled`。  
     - `supported`：早段外部 RH 上升时，内部 RH 同步跟随；但这种响应没有在后段继续异常持续，因此更像正常的外部驱动耦合。  
     - `breathing_watch`：早段同样会跟随，但后段这种耦合仍然持续，且 `AH 衰减 / 驱动力` 比例接近 0，更像 `材料呼吸 / 释湿` 对信号的污染。  
   - 这一步没有改变当前三态结论，但把 `supported / negative / breathing_watch` 从“多尺度规则结果”进一步落成了更容易解释给老师和工程侧的物理语言。
   - 在此基础上新增了 `cooling residual probe v1`，专门验证“扣除纯降温导致的 RH 理论回升后，冷却段是否还有额外 AH 增益”：  
     - 当前结论不是“冷却段无效”，而是“现有 candidate 更多表现为 `dAH` 相对改善，而不是稳健的外部进湿残差”。  
     - 现有最强的 `unsealed heated` 候选在残差视角下仍然只有 `weak_or_confounded`，并且所选冷却段里 `外部 AH 余量` 为负，这说明当前 `dAH` 变好看并不能直接解释成外部湿气进入。  
     - 因此，冷却段当前仍应保留为 `物理解释探针 + review 辅助证据`，不能提前升成默认主判定分支。
   - 进一步收紧为 `cooling residual probe v2` 之后，结论变得更明确：  
     - `v2` 改成了 `onset 优先 + 外部 AH 余量硬门控`，只有“最早出现且外部确实有正向绝对湿度余量”的冷却块才允许进入后续残差判断。  
     - 在当前数据上，唯一的 `unsealed heated` 冷却候选也被收敛成了 `no_external_headroom`，说明现有冷却块并不满足“外部高湿空气可进入”的物理前提。  
     - 这一步把问题从“冷却段阈值没调好”收敛成了“当前冷却段没有形成可验证的外部进湿条件”；因此这条线后续不该靠调阈值推进，而应等待更清晰的冷却参考段或回到 `无热源高湿` 主战场继续优化。

8. **`evidence fuser v2` 已通过本轮离线验收，但其含义是“运行级决策结构通过”，不是“问题已被统一分类器解决”**  
   - 当前 `v2` 已把三类新状态真正接入运行级输出：`transition_boost_alert`、`static_dynamic_support_alert`、`static_hard_case_watch`。  
   - 它的核心收益不是新增一个黑盒模型，而是把多视角证据变成更稳的工程行为：转移证据增强、静态漏报补回、静态难例主动保守。  
   - 当前离线验收结果显示：`transition_boost_capture_rate = 1.0`，`static_eval_balanced_accuracy = 1.0`，`static_prediction_coverage = 0.667`。  
   - 这意味着当前最适合现场演示的不是“全自动 seal/unseal 分类器”，而是“强证据直接进入 review，难例和干扰主动降级”的决策系统。

9. **`evidence fuser v4` 已完成首轮对照验证，并可作为当前默认路径替换 `v3`**  
   - `v4` 的升级重点仍然只放在数据已经明确支持的部分：保留 `gate / info selector v3` 与 `transition` 多视角分数不变，只把 `no_heat probe v3` 以极小范围接回 `外部高湿-无热源` 静态支线。  
   - 对照结果显示：`status_changed_count = 1`，且唯一变化文件就是 `2026-03-03 181049_unseal_unheated`；它从 `static_dynamic_support_alert` 升级成了 `static_dynamic_supported_alert`，而所有 transition 运行与其他分支都没有回退。  
   - 同时，`v4` 继续保持 `transition_capture_rate = 1.0`、`transition_boost_capture_rate = 1.0`、`static_eval_balanced_accuracy = 1.0`、`static_prediction_coverage = 0.667`，并通过了 “变更仅限 no-heat 主战场” 的保守验收。  
   - 因此当前主路线应正式以 `evidence fuser v4` 作为默认实现，而不是继续停留在 `v3`。

10. **`transition event summary v1` 已完成首轮验证，可直接服务现场演示**  
   - 这一步不引入新模型，只把 `transition_score_v3` 进一步压成 `event_start / event_end / peak_time / duration` 这类事件级输出。  
   - 当前两个 transition 运行都能提取出单一主事件，且 `event_detect_rate = 1.0`、`near_overlap_rate = 1.0`、`peak_in_near_rate = 1.0`。  
   - 这意味着现场展示时，不再需要手工解释一串窗口分数，而可以直接讲“事件从何时开始、何时达到峰值、何时回落”。  
   - 当前演示入口也已同步到这一路径，因此现场默认口径应理解为：`分支感知路由 -> 转移段相对打分 -> 状态段级证据融合 v4 -> transition event summary`。

11. **统一数据接口已落地，可作为后续现场迁移与复核闭环的基线**  
   - 当前已经实现 `Unified Data Interface v1`，并已切换到 `evidence fuser v4` 作为默认来源，把现有主路线统一导出为三张正式表：`run_manifest`、`window_table`、`review_output`。  
   - `run_manifest` 负责描述运行级元数据和当前决策状态；`window_table` 负责沉淀窗口级路由、`segment_id`、`info_score` 和粗粒度标签；`review_output` 负责沉淀待复核队列及其关键证据。  
   - 这一步的价值不是提高实验室指标，而是把当前已验证的决策链收口成可复用的数据接口，便于后续接入现场数据、历史健康窗口库和人工复核结果。  
   - 因此，后续所有迁移与增量实验都应优先基于这三张表继续推进，而不是继续从零拼接不同报告。

12. **`health window bank + similarity ranking v1` 已开始落地，但当前定位必须收紧**  
   - 当前已经基于 `run_manifest / window_table / review_output` 建立了首版健康窗口库和相似性风险排序接口。  
   - 它当前更适合作为“现场迁移骨架”和“复核优先级排序器”，而不是新的实验室主判定器。  
   - 当前健康库只有 `21` 个窗口、`3` 个健康参考运行，规模仍小，因此这条线应理解为：先证明接口、分层和排序流程能跑通，再等待后续现场历史窗口逐步扩库。  
   - 这也解释了为什么当前这条线不以 `seal/unseal` 分类指标为主，而以“能否把新查询相对健康库做偏离排序”作为当前目标。

13. **健康窗口库的复核回灌闭环已经打通**  
   - 当前已经实现 `Health Bank Feedback Loop v1`，可以把 `review_output` 中的人工复核结果回灌成三类动作：  
     - `healthy`：提升为健康参考，增量加入健康窗口库；  
     - `anomaly`：沉淀为异常参考；  
     - `uncertain`：保留在待复核队列。  
   - 首轮闭环验证表明：即使只补少量人工复核结果，也能把健康库从 `3` 个参考运行扩到 `4` 个，并对剩余 `pending` 运行重新做相似性排序。  
   - 这一步的意义不是提高实验室单次指标，而是证明后续现场历史窗口库可以通过“运行级复核 -> 窗口级增量入库”的方式逐步成长。  
   - 因此，后续现场迁移最合理的工程路径已经明确：不是先上更复杂模型，而是先把这条 `review -> bank update -> rerank` 的闭环持续跑起来。

## 5.3 第三阶段：融合层与更复杂模型

只有当同一工况族里至少两个分支表现出明显互补性，才进入更复杂的模型融合层：

- 默认先用 `Logistic Regression` 或 `LightGBM` 做轻量融合。
- `stacking / gating / boosting / MoE` 放到第三阶段，不作为当前首轮主线。
- 如果第一阶段都没有找到高信息工况，第三阶段直接取消，回到实验设计。
- 当前已经实现并采用为默认路径的 `evidence fuser v4` 属于**规则化证据融合 / 决策层**，不属于这里所说的“更复杂模型融合层”。

## 5.4 本轮实验组织方式

所有实验统一按下面这个维度组织：

`工况族 × 窗口大小 × 步长 × 分支模型`

验证方式固定为：

- 按实验运行批次做 `leave-one-run-out`
- 不允许把同一运行批次里的相邻窗口同时放进训练和测试
- 第一轮只输出三类结论：
  - 结论 A：哪些工况应直接排除
  - 结论 B：哪些工况存在稳定可分性
  - 结论 C：状态转移时分数能否在变化区域附近持续抬升

## 6. 后续现场迁移

现场迁移不再定义为当前阶段的主要任务，但路线应提前约束好：

1. 先把现场任务定义为**风险排序 / 相似性评分**，不是直接硬分类。
2. 先做**设备内历史健康窗口库**，再做相似性检索与残差偏离。
3. 现场阈值必须依赖小批量人工复核做校准，而不是直接沿用实验室阈值。
4. 若实验室阶段没有建立稳定可分性证据，不应直接推进大规模现场上线。

## 7. Go / No-Go 标准

## 7.1 Go 到第二阶段的条件

满足以下条件之一，才进入三分支建模：

- 至少一个高信息工况族在 `leave-one-run-out` 验证下显示出重复性的区分或排序能力。
- 状态转移场景中，异常分数能在多数转移样本的邻域附近持续抬升。

## 7.2 Go 到第三阶段融合的条件

只有同时满足下面两点，才讨论融合层或更复杂模型：

- 至少两个分支在同一工况族内表现出互补性；
- 轻量融合已经显示出比单分支更稳定的收益。

## 7.3 No-Go 条件

若首轮没有任何高信息工况族通过验证，则下一步不是继续换模型，而是：

1. 回到实验设计；
2. 增加更强激励或更清晰的状态切换实验；
3. 重新定义工况边界与标注方式。
