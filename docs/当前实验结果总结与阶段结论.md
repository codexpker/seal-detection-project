# 当前实验结果总结与阶段结论

> 这份文档只基于当前已经完成的实验、脚本和报告做总结，不引入新实验，也不引入超出现有结果的新结论。  
> 目标是回答 3 个问题：  
> 1. 当前到底做到了什么；  
> 2. 哪些地方已经可以支撑汇报和演示；  
> 3. 哪些地方仍然不能被表述成“已经解决”。

---

## 1. 一句话阶段判断

当前项目已经证明：

**“分支感知路由 -> transition 事件证据 -> 静态弱证据 -> watch / abstain / review” 这条路线是成立的。**

但当前项目还没有证明：

**“在大多数工况下，单一统一模型可以稳定完成密封 / 非密封全局分类”。**

因此，现阶段的正确定位不是“统一分类器已经成功”，而是：

**已经形成了一个可解释、可保守、可复核、可逐步迁移到现场的证据驱动决策系统。**

---

## 2. 当前主路线是什么

当前默认主路线已经固定为：

`gate / info selector v3 -> evidence_fuser v3 -> transition event summary`

这条路线的核心含义不是“所有数据都强行二分类”，而是：

1. 先做分支感知路由，只让高价值窗口进入后续分支；
2. 对状态转移场景，用事件级证据做强判断；
3. 对静态场景，只给辅助风险，不承诺全覆盖；
4. 对干扰样本和难例，主动进入 `watch / abstain`，避免乱报。

---

## 3. 当前已经被实验结果证明的能力

## 3.1 Transition 场景已经做出来了

这是当前最强、最可信的能力。

根据 `transition_event_summary_v1`：

- `transition_files = 2`
- `detected_events = 2`
- `event_detect_rate = 1.0`
- `near_overlap_rate = 1.0`
- `peak_in_near_rate = 1.0`

这说明：

1. 当前系统不仅能在 transition 场景中抬分；
2. 而且能把分数压成事件结构；
3. 并且事件峰值落在关键转移邻域内。

也就是说，系统已经不是“只能看到单窗异常”，而是已经能输出：

- `event_start`
- `event_end`
- `peak_time`
- `duration`

这使得 transition 场景已经具备现场演示和后续人工复核的基础。

---

## 3.2 运行级证据融合链已经成型

根据 `phase3_evidence_fuser_v3`：

- 结论：`PASS`
- `transition_capture_rate = 1.0`
- `transition_boost_capture_rate = 1.0`
- `static_eval_balanced_accuracy = 1.0`
- `static_prediction_coverage = 0.667`
- `adopt_v3_default = True`

这组结果说明：

1. 当前系统已经能把不同证据组织成运行级结论；
2. Transition 分支没有回退，反而在解释强度上更好；
3. 静态分支已经不是“随便判”，而是“有证据才说，没有把握就 abstain/watch”。

换句话说，当前系统最接近的定位不是“分类器”，而是“运行级风险决策器”。

---

## 3.3 干扰场景下的保守性是成立的

根据 `old_data` 干扰回归测试：

- `strict sealed = 19`
- `strict_safe_rate = 0.579`
- `strict_watch_rate = 0.421`
- `strict_positive_like_count = 0`

这一步非常关键，因为它说明：

1. 严格密封的历史高干扰样本，没有被直接打成泄漏；
2. 当前系统在“看不准”的场景下，会优先落到 `safe` 或 `watch`；
3. 这说明系统具备基本工程可信度，而不是一味追求激进告警。

对于现场演示来说，这一点和 transition 检出能力同样重要。

---

## 3.4 现场迁移的工程骨架已经搭好

当前已经实现了统一数据接口：

- `run_manifest`
- `window_table`
- `review_output`

同时已经实现了：

- 健康窗口库
- 相似性风险排序
- 人工复核后回灌健康库

这意味着当前项目不再停留在“实验室里跑几个脚本”，而是已经具备了从实验室过渡到现场的最小工程结构。

现阶段这条线的价值主要是：

1. 统一后续数据组织方式；
2. 让后续现场数据可以直接接入；
3. 让人工复核结果能沉淀成健康参考和异常参考；
4. 让相似性排序逐步从实验室骨架扩展到设备级历史窗口库。

---

## 4. 当前实验结果真正说明了什么

如果只基于当前结果，最准确的结论是：

## 4.1 已经证明“证据驱动路线”是对的

当前结果已经支持以下判断：

1. 先做工况筛选 / 路由，而不是直接做全局分类，是正确的；
2. Transition 场景比静态全局判别更容易形成稳定证据；
3. 静态场景必须允许 `watch / abstain`，不能强推全覆盖；
4. 历史旧数据更适合作为干扰挑战集，而不是直接并入主训练集；
5. 现场迁移更适合从“健康窗口库 + 相似性排序 + 人工复核回灌”开始，而不是先上复杂深度模型。

## 4.2 还没有证明“统一分类器已经成功”

当前结果不支持以下表述：

1. “已经解决密封 / 非密封统一判别”
2. “静态 seal / unseal 在大多数工况下已经稳定可分”
3. “相似性库已经足够大，可以单独替代主链路”
4. “旧数据可以直接作为统一监督训练集”

如果在汇报里直接把项目表述成“已经做成统一分类器”，会明显超出现有证据范围。

---

## 5. 当前最重要的难点和边界

## 5.1 静态场景仍然不是统一可分

多视角分析已经说明，静态场景里真正值得追的不是原始曲线和几个简单量，而是：

- 动态响应相关性
- 高分位窗口特征
- 阈值 favored 持续性
- 转移邻域的抬升速度

这本身就说明：

当前问题不是“多训一个分类器就能解决”，而是静态可分性本来就弱。

## 5.2 存在明确的密封难例

当前最典型的难例是：

- `2026-03-06 160246_seal_unheated`

它之所以关键，是因为它说明：

1. 某些严格意义上的密封运行，也会在多视角特征空间里靠近非密封运行；
2. 如果系统没有 `hard_case_watch`，很容易误报；
3. 因此“强制二分类”在当前阶段是有真实工程风险的。

## 5.3 健康窗口库现在还偏小

当前健康库只有少量健康参考运行和窗口，因此：

1. 相似性这条线现在更像“迁移骨架”；
2. 它适合做排序和复核辅助；
3. 还不适合作为当前实验室主判定器。

---

## 6. 当前最适合的对外口径

如果需要给老师、组里或现场演示做总结，建议统一使用下面这套口径：

1. 当前不再把问题定义为“做一个全局统一密封 / 非密封分类器”；
2. 当前系统先做分支感知路由，再分别处理 transition、静态和干扰场景；
3. Transition 场景已经能够稳定输出事件级证据；
4. 静态场景当前只作为辅助风险，不承诺全覆盖；
5. 对干扰样本和难例，系统会主动进入 `watch / abstain`，而不是乱报；
6. 当前已经具备统一数据接口、健康窗口库和复核回灌闭环，为后续现场迁移做好了骨架准备。

---

## 7. 现阶段最值得继续做的事情

在不考虑新实验的前提下，当前最值得做的不是继续发明新模型，而是：

1. 用真实人工复核结果替换演示标签；
2. 继续扩充健康窗口库；
3. 把后续数据都统一接入 `run_manifest / window_table / review_output`；
4. 继续把“历史健康窗口库 + 相似性排序 + 人工复核回灌”这条线做稳；
5. 保持当前主路线不变，不回退到“全局 seal / unseal 分类器”思路。

---

## 8. 阶段结论

当前实验结果已经足够支持以下阶段结论：

**项目已经从“实验室探索”推进到“可解释决策链 + 复核闭环 + 现场迁移骨架”的阶段。**

当前最强能力是：

- transition 事件检测
- 运行级证据融合
- 干扰场景下的保守决策

当前仍需保持克制的地方是：

- 静态 seal / unseal 统一判别
- 小样本下的全局分类结论
- 健康窗口库尚未扩展到设备级现场规模

因此，现阶段最准确的项目表述应该是：

**已经证明路线正确，已经形成可演示、可复核、可迁移的工程主线；但还没有证明统一分类问题已经彻底解决。**
